# 缓存命中优化 — NΞXUS 实战指南
> 创建: 2026-06-20 · 迭代: 第1轮 · 作者: Hermes (mimo-v2.5-pro)

## 核心原理

LLM 缓存有三层：

```
请求 → 语义缓存(100%省) → 前缀缓存(90%省) → 完全推理(全价)
         ↓                    ↓                    ↓
    返回缓存响应          返回缓存KV状态         新鲜计算
```

### MiMo 缓存价格

| 类型 | 价格 | 相比全价 |
|------|------|---------|
| 缓存未命中（新鲜计算） | ¥3/百万 token | 100% |
| 缓存写入（首次） | 免费（限时） | 0% |
| 缓存命中（读取） | ¥0.025/百万 token | **0.8%** |
| 缓存 TTL | 1 小时（持续命中可延长） | — |

> **关键洞察**：MiMo 缓存命中价格是 DeepSeek V4 Pro 全价的 **1/70**。

## 前缀缓存优化（最关键）

前缀缓存的原理：如果多个请求的开头部分（前缀）相同，模型只需计算一次 KV 状态，后续请求直接复用。

### 规则1: 静态内容前置，动态内容后置

```
[系统提示 — 静态，不变] ← 缓存点1 ✓
[工具定义 — 几乎不变]  ← 缓存点2 ✓  
[历史消息 — 只追加]    ← 缓存点3 ✓
[最新用户消息 — 每次变] ← 不缓存
```

**NΞXUS 实践**：
- Hermes 的系统提示（nexus-core + memory）约占 3-5K tokens，每次请求都相同
- 工具定义约占 2-3K tokens，几乎不变
- 这两部分加起来 5-8K tokens，如果每次都缓存命中，每次请求节省 ¥0.015

### 规则2: 保持提示结构一致

**正面**：系统提示结构稳定，Skill 按固定顺序加载
**负面**：随机拼接内容、动态改变提示结构会破坏缓存

### 规则3: 会话历史只追加不修改

LCM 压缩会改变历史内容，导致缓存失效。解决方案：
- LCM 开启 `CACHE_FRIENDLY_CONDENSATION=true`
- 压缩时保留前 N 条不变，只压缩中间部分

### 三断点架构

```
[系统提示] —BP1→ [工具定义] —BP3→ [历史消息] —BP2→ [最新消息]
   静态           几乎静态        只追加          每次变
   1小时TTL       1小时TTL       5分钟TTL       不缓存
```

**NΞXUS 映射**：
- BP1：nexus-core Skill + memory（静态，小时级不变）
- BP3：工具定义（hermes 工具集，几乎不变）
- BP2：会话历史（LCM 管理，只追加）
- 动态：最新用户消息

## CC (Claude Code) 的缓存

CC 的缓存行为：
- 每次任务是独立会话，缓存无法跨任务复用
- 但任务内的多轮对话有前缀缓存
- 系统提示（CLAUDE.md）+ 工具定义是静态前缀

**优化**：
- CLAUDE.md 保持精简（减少每次请求的基础 Token）
- 任务描述放在 prompt 而不是 CLAUDE.md 里

## 语义缓存（应用层）

对于重复性高的查询（如定时报告），可以在应用层做语义缓存：

```
查询 "BTC 今日行情" → 哈希匹配 → 返回缓存结果（零 Token）
查询 "BTC 最新走势" → 语义匹配 → 同上（零 Token）
查询 "帮我分析 BTC 趋势" → 不匹配 → 调用 LLM
```

**NΞXUS 适用场景**：
- no_agent Cron 任务（脚本输出本身就是缓存）
- 高频重复查询

## 监控缓存命中率

```bash
# 检查最近一次 CC 调用的缓存
# 从 CC 返回的 JSON 中读取 cache_read_input_tokens
```
