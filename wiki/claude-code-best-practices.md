## 三阶段工作流

### 阶段 ①：先问答，熟悉代码库
- 接到任务先理解，不要上来就改代码
- 查类用法、查函数签名、查 Git 历史
- 用 `search_files` 找相关代码，用 `git log` 查变更历史
- **目的**：搞清楚"这段代码为什么这样写"

### 阶段 ②：先规划，后执行
- 不要直接实现庞大功能
- 先出方案/计划 → 等确认 → 再动手写代码
- 拆解为小步骤：每步 1-2 个文件，改完验证再继续
- **NΞXUS 对应**：四阶段防作弊流程（需求理解→拆解分发→执行验证→终验报告）

### 阶段 ③：执行 + 工具验证
- 写代码 → 调测试 → 截图验证 → 迭代优化
- 能调 CI 就调 CI，能写单测就写单测
- 每次改动后立即验证，不要等全部完成再检查

## CLAUDE.md 层级结构

```
~/.claude/CLAUDE.md          ← 全局（每次启动自动加载）
项目根目录/CLAUDE.md         ← 项目级（进入项目时加载）
项目根目录/local.claude.md   ← 个人级（不提交 Git）
子目录/CLAUDE.md             ← 子目录级（在该目录工作时自动加载）
```

**层级优先级**：子目录 > 项目级 > 全局（越近越优先）

## 工具集成

CC 能调用外部工具扩展能力：
- **CI 工具**：调 CI 验证代码
- **MCP 工具**：接入团队内部工具
- **测试**：写单测、截图迭代
- **Git**：自然语言指令（"提交推送 PR"）

## 自然语言 Git 指令

| 指令 | CC 自动执行 |
|------|------------|
| `提交推送 PR` | commit + push + 创建 PR |
| `查看最近变更` | git log + git diff |
| `创建新分支` | git checkout -b feature/xxx |
| `合并主分支` | git merge main |

## 与 NΞXUS 四阶段的对应

| 官方建议 | NΞXUS 四阶段 |
|---------|-------------|
| 先问答熟悉代码 | 阶段 0：需求理解 |
| 先规划后执行 | 阶段 1：拆解分发 |
| 执行+工具验证 | 阶段 2：执行验证 |
| 迭代优化 | 阶段 3：终验报告 |
