## 🎯 WHAT — 核心问题与原理

双 Agent 协作的致命陷阱是**责任模糊**：Hermes 觉得 CC 会做，CC 觉得 Hermes 会做，结果都不做。或者更糟——两个都声称做了但实际没做（幻觉协作）。

**解决方案**：用"产出类型"一刀切划清边界，不留灰色地带。

### 判断公式（一句话裁决）

> **产出 = 代码变更 → CC | 产出 ≠ 代码变更 → Hermes**

## 🛠️ HOW — 关键实现

| 工具链安装 | ✅ | ❌ |
| 指挥 CC | ✅ 派活、验收 | ❌ |

### 4 阶段工作流

```
阶段 0：需求理解（Hermes，强制回环）
  ├─ 5 问必须全部回答：
  │  ① 要达成什么结果？
  │  ② 验收标准是什么？
  │  ③ 不做什么？
  │  ④ 依赖是什么？
  │  ⑤ 优先级/风险？
  └─ 不全 → 追问用户，不发任务

阶段 1：任务拆解 + 分发（Hermes）
  ├─ 标注 [CC] 或 [ME]
  └─ 每个 CC 任务附带：目标/验收/物证要求

阶段 2：执行 + 三重验证（不可跳过）
  ├─ CC 产出验证：
  │  ① 物证核对（文件存在？内容对？）
  │  ② 功能验证（跑测试/看输出）
  │  ③ Git 溯源（commit 存在？diff 合理？）
  └─ Hermes 自我约束：
      操作留痕、不做无验证结论

阶段 3：终验报告（Hermes → 用户）
  ├─ 实际做了什么（事实，非计划）
  ├─ 验证结果（通过的证据）
  └─ 没做的 / 遗留项
```

## 🔄 APPLICATION — Hermes 集成方案

### CC 任务模版

每次派发给 CC 时使用：

```
任务：<一句话>
目标：<可验证的结果>
物证：<必须返回的证据（文件路径、测试输出等）>
禁止：<明确不能做的事>
```

### 防作弊规则

| 规则 | Hermes | Claude Code |
|------|--------|-------------|
| 虚构文件内容 | ❌ | ❌ |
| 声称测试通过但没跑 | ❌ | ❌ |
| 声称已修复但没改 | ❌ | ❌ |
| 跳过验收 | ❌ | ❌ |
| 不追问就分发 | ❌ | N/A |
| 虚构命令输出 | ❌ | N/A |
| 抄 CC 的话当验证 | ❌ | N/A |

### 启动 CC

```bash
start-claude
```
